My housemate got in me into MTG when he promised it was like a game I adored as a kid, Pokemon cards.

The more established websites provided ways to practice certain elements of the game like drafting. However, when new cards were released they were often very slow to update their websites. I believed this could achieved by creating an easily updated site.

The website gets alot of hits, especially around the release of a new expansion (35,000 in September 2014). The audience is from all over the world but the vast majority come from the US. The website feels faster than any of the other similiar sites as its processed via client side JavaScript.

What does it do?

The application consists of 4 parts.

Draft Simulator

In the draft simulator you play against 7 other AIs, taking it in turn to pick cards. The AI uses a fairly simple strategy for picking based on the rating of the cards and the 'colors' (teams) the cards belong to.

When the all the cards are assigned a player are able to build a deck from them. The app uses chartjs to generate graphical analysis of the cards the player has included in their deck.

Sealed Simulator

Another popular way of playing Magic is 'Sealed' play where each player opens a number of packs and tries to build a deck from the contents.

Like the draft simulator my website provides graphical analysis of the cards picked and allows the user to download their selection as a text file so they can import their selection to other programs.

Other parts

The site also features a booster simulator and a card browser which were mainly created through reusing code from the parts above.

Choice of technology

The majority of the website is built using client side Javascript. This keeps the load on the server smaller and utilises the power of modern devices. I used AngularJS. Much of the logic for opening packs etc was shared between the different views so using Angular allowed me to put all this logic into services which could be utilised by any of the views.

I used Bootstrap to make the website responsive. The site adapts nicely to whatever device you're viewing it on.

I was also able to include some other sweet technologies like HTML5's Application Cache and File Saver.

Challenges Faced

The biggest challenge faced was trying to get the website working as quickly as possible. I needed to show the card images for many cards at once and I new it would be a pain if each time a user opened a pack they had to wait for all these images to load.

The solution to this was to use HTML5's Application Cache. This meant I could download all the images from the latest set to the user's device as soon as they go to the website. Once this happens once the images never need to be loaded again (as long as the user doesn't clear the app data). There was another cool side effect of this - once all the data was cached the app is usuable whether you are online or not.

My other challenge was trying reduce the clutter on the interface. I have tried to hide more advanced options within toggles and kept non-essential buttons small while making frequently used buttons clear and large. This keeps it simple for the average user while allowing further customization for those who want it.
